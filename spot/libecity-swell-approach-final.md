## 前提の再認識：今の自分はどこにいるか

全否定が明らかにした最大の事実は、**訴求以前の問題** がある、ということ。

```
訴求の良し悪し以前に
「誰が言っているか」が何より先に問われるコミュニティ
```

リベシティは情報の正しさより **人への信頼** が先に来る。
どれだけ訴求が正確でも、存在が見えない人の発信はスルーされるか疑われる。

**今やるべきは「訴求の最適化」ではなく「存在の可視化」。**

## フェーズ設計：3段階で積み上げる

### Phase 0：存在を知ってもらう（期間：1〜2ヶ月）

**目標：「この人、SWELLやってる人だ」と認識されること。**
サービス訴求・TIPS・宣伝はまだしない。

#### やること
- **体験談の等身大発信**
  「SWELL-DPを作っている」という事実を、TIPS形式ではなく日記・進捗報告として出す
  ```
  NG：「SWELLのコピペ方法を教えます」
  OK：「SWELL-DPというコピペパターン集を作ってます。
       今日ヒーローセクションが完成しました。
       ここのレイアウト、何度試しても崩れてたんですが…」
  ```
- **他の人の投稿に先に反応する**
  SWELLの話題が出たら、売り込みなしで「わかります」「自分も詰まった」と反応する
  → 「この人、SWELLに詳しい」という印象を積み上げる

- **失敗・詰まりを出す**
  「うまくいった」より「詰まってこう解決した」の方がリアリティがあり信頼される
  全否定で明らかになった「ポジティブバイアス」の裏を突く

#### KPI
- 特定の投稿に1〜3人の反応がつく
- 「SWELLの人」として認識されるコメントをもらう

---

### Phase 1：価値を見せる（期間：2〜4ヶ月）

**目標：「何を作っているか」が具体的に伝わること。**
サービスへの誘導はまだしない。「見たい」と思ってもらうことが目的。

#### やること
- **ビフォーアフターを見せる**
  「このセクション、SWELLで組むとどうなるか」のスクリーンショット投稿
  コードの話より **見た目・完成形** で興味を引く
  ```
  「SWELLでお客様の声セクションを作ったら
   こうなりました。ブロックコードのコピペだけです。
   （スクショ）
   詳しくはプロフのリンクから」
  ```

- **具体的なユースケースを1つに絞る**
  全否定が示した通り「広く刺さろう」とすると薄くなる。
  最初は **「LP制作をSWELLで受けている制作者」** に絞って話しかける
  → ブロガーや初心者への展開は後回し

- **「試せる」場所をシンプルに示す**
  「コピペ完結」という言葉は使わず「無料で試せます」に変える
  URLは毎回貼らず、「気になったらDMください」「プロフに置いてます」で十分

#### KPI
- 投稿に「どこで使えますか？」「URL教えてください」のリアクションが出る
- サイトへのアクセスが自然流入で発生する

---

### Phase 2：訴求する（期間：4ヶ月目以降）

**目標：「使ってみたい」から「使った」に変える。**
Phase 0〜1で積み上げた信頼の上に、初めて訴求が乗る。

#### やること
- **ペルソナ分析の「仮説」を実データで上書きする**
  - Phase 0〜1の反応で「実際に刺さった言葉・投稿」をメモしておく
  - 反応が多かった投稿 = 本当に刺さったペイン
  - これが「実証済み訴求角度」になる

- **訴求は「最も反応が出たペイン」だけに絞る**
  分析で出てきた複数の角度を同時に使わない
  1投稿1メッセージ、1メッセージ1ペイン

- **「証拠」を持って話す**
  - 利用者の声（たとえ1人でも）
  - 制作時間の比較（自分の体験ベースでも）
  - 「○ヶ月で○パターン作った」という実績の数字

#### KPI
- サイトへのアクセス増加
- コピー使用・問い合わせの発生

## 訴求の「言葉」設計

Phase 1〜2で使う言葉は、ポジティブ分析・全否定の両方を通過させて選ぶ。

### 通過基準
1. ブロガー層に「何の話？」と思われないか
2. 「簡単・すぐできる系の煽り」に聞こえないか
3. 知名度ゼロでも宣伝臭がしないか
4. 半年後に陳腐化しないか

### 通過した言葉・フレーム

| 使える | 使えない |
|--------|---------|
| 「SWELLで作ってるLPのセクション、こうしてます」 | 「コピペ1分で完成！」 |
| 「このセクション、どう組みますか？私はこうしました」 | 「AI専用ブロック問題を解決！」 |
| 「無料で試せます」 | 「制作時間が半分に！」 |
| 「SWELLのブロック機能だけで作りました」 | 「人間が設計した本物のコード」 |
| 「LP制作を受けている方、こういう使い方もあります」 | 「SWELLユーザー全員に刺さります」 |

## 「整う」ために見えてきた本質

ポジティブ分析と全否定を経て残ったのは、1つのシンプルな構造だった。

```
信頼（人）→ 興味（見た目）→ 理解（使い方）→ 行動（試す）
```

この順番を**飛ばすことはできない。**
ほとんどの失敗は「信頼」を飛ばして「行動」を求めるから起きる。

ペルソナ分析で明らかになった「コピペへの欲求」も、
全否定で指摘された「コピペへの不信感」も、
どちらも本当のことで、矛盾していない。

```
「コピペしたい」←→「コピペは怖い」

この矛盾を解くのは「訴求の言葉」ではなく
「信頼できる人が言っているかどうか」だった。
```
