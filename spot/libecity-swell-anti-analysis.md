# SWELL-DP リベシティ訴求 全否定レポート
# ─ 失敗するパターンと解決策

> 作成日：2026-04-02
> 前提：`libecity-swell-persona-analysis.md` の分析結果を意図的に全否定し、
> 「この訴求でなぜ失敗するか」を先回りして潰すための後学資料。

---

## はじめに

ペルソナ分析は「刺さる」を探す作業だが、
その裏には必ず「刺さらない理由」が潜んでいる。
この資料では **分析結果のポジティブな解釈をすべて否定し、**
失敗シナリオと対処を先に言語化しておく。

---

## 否定1：「AIがSWELL専用ブロックを使わない問題への回答」は刺さらない

### なぜ失敗するか

「AIがSWELL専用ブロックを使ってくれない」と発言したのは **投稿全体のごく一部（2〜3名）** に過ぎない。
分析では「一番刺さる」と結論づけたが、それは **少数の声を過大評価したサンプリングバイアス** である。

リベシティの大多数（特にブロガー・初心者層）は：
- そもそもSWELL専用ブロックと標準ブロックの違いを意識していない
- AIでWordPressを実装しようとすらしていない
- 「コードを貼る」行為自体に心理的ハードルがある

「AIとSWELL専用ブロックの相性問題」は **上級者向けの課題** であり、
多数派には「何の話をしているのかわからない」で終わる。

### 失敗パターン
```
「AIがSWELL専用ブロックを使ってくれない問題、解決しました。」

→ ブロガー層の反応：「え？そんな問題があったの？」
→ 初心者層の反応：「SWELL専用ブロック……？」
→ 結果：ニッチすぎてスルーされる
```

### 解決策
- 訴求の「入口」を「SWELLをもっとキレイに使いたい人へ」に広げる
- AI文脈は「補足説明」に留め、主訴求には使わない
- 「専用ブロック」という技術用語を使わず「SWELLのブロック機能だけで作れる」と言い換える

---

## 否定2：「コピペ完結」は信用されない

### なぜ失敗するか

「コピペするだけで完成」という訴求は、リベシティユーザーには **「どうせうまくいかない」という経験則** がある。

多くのユーザーが過去に：
- 「簡単です」と言われたが実際は環境依存でエラーが出た
- CSSをコピペしたら他のところが崩れた
- プラグインを入れたらサイトが壊れた

という体験をしており、**「コピペ完結」は疑われるフレーズ** になっている。
また「簡単・すぐできる」系の訴求はリベシティ内では **情報商材・煽り系のトーン** に近く、信頼を損なうリスクがある。

### 失敗パターン
```
「コピペ1分でLPセクション完成！」

→ 「どうせ環境によって違うんでしょ」
→ 「どこかで詰まるはず」
→ 「売り込みっぽい」
→ 結果：読まれても信用されない
```

### 解決策
- 「できる」より **「こういう条件下でできる」** を正直に書く
  - 例：「SWELLテーマ＋SWELL標準ブロックを使用している環境であれば」
- 「コピペで完成」より **「コピペで試せる」** という表現に変える
- 動作確認済みであることを示す（「Local環境で動作確認済み」など）
- 失敗しやすいポイントをあえて先出しすることで信頼感を上げる

---

## 否定3：「#SWELLメモ 形式に寄せる」は逆効果になる可能性がある

### なぜ失敗するか

`#SWELLメモ` で投稿しているのは **常連の特定ユーザー（主にけん氏）** の個人スタイルであり、
他ユーザーが同じ形式で投稿しても **「模倣・乗っかり」** に見える可能性がある。

リベシティのコミュニティ文化は：
- 長期間積み上げた信頼関係が前提
- 新参者が既存スタイルを真似ると「誰これ」になりやすい
- 価値提供より先に **「何者か」が問われるコミュニティ**

さらに、TIPS形式の投稿は「教える人」のポジションを要求するため、
知名度がない段階での発信は **「上から目線」に映るリスク** がある。

### 失敗パターン
```
（知名度ゼロの状態でいきなり）
「SWELLでLPセクションをコピペする方法 👀
① cn-design.biz でパターンを選ぶ
② コピーボタンを押す
#SWELLメモ #SWELL」

→ 「宣伝じゃん」
→ 「誰？」
→ 反応なし・フォロワー増えない
→ 結果：スパム扱いに近い印象を与える
```

### 解決策
- **TIPS発信より先に「共感・体験談」で存在を知ってもらう期間を設ける**
- 「教える」より「一緒にやってみた」「失敗した」「こうしたら上手くいった」という **等身大の実践報告** から入る
- サービス紹介は最後の1行に留め、最初は価値提供より **人物像の共有** を優先する
- 発信の順番：体験共有 → 反応・関係構築 → TIPS → サービス紹介

## 否定4：「制作スピード向上・時短」は差別化にならない

### なぜ失敗するか

「制作時間を短縮できる」「効率化できる」は **あらゆるサービス・ツールが言っている** 訴求であり、
リベシティ内でも類似の文脈の発信が溢れている。

具体的な競合：
- Codex / Cursor / Claude Codeによる自動実装
- SWELLのテンプレート機能・パターン機能
- 他のWordPressブロックパターンライブラリ
- ブロックエディタのコピー機能（そもそも自作パターンをストックできる）

「時短」「コピペ完結」だけでは **なぜSWELL-DPでなければならないのか** が伝わらない。

### 失敗パターン
```
「SWELL×コピペで制作時間を半分にした方法」

→ 「そういうツール他にもあるよね」
→ 「自分でパターン作ればよくない？」
→ 「SWELLのパターン機能じゃダメなの？」
→ 結果：差別化できず埋もれる
```

### 解決策
- **「なぜSWELL-DPか」の独自性を一言で言えるようにする**
  - 例：「SWELLのパターン機能との違いは○○」を明言する
  - 例：「自作するより○○な理由」を具体的に示す
- 時短を訴求するなら **「何時間が何分になるか」の定量化** が必要
- 「LP1本分のデザイン選定・コーディングが○時間→○分」など実体験ベースの数字を出す

## 否定5：「人間が設計した」という差別化は今後通用しなくなる

### なぜ失敗するか

「AIが作っただけのサイトじゃない」という訴求は現時点ではポジションが取れるが、
**AI生成の品質が急速に上がっている** ため、半年〜1年以内に陳腐化する可能性が高い。

「人間設計 vs AI生成」という対立軸は：
- AIアレルギーのある保守層には響くが、AI活用に積極的な層には刺さらない
- リベシティはAI活用に前向きなコミュニティのため **AIを否定する文脈はマイノリティ向け**
- 「人間が作った」は品質担保にはなるが、それ自体が購買動機にはなりにくい

### 失敗パターン
```
「AIではなく、人間が設計したブロックコード」

→ AI活用ユーザー：「別にAIでいいじゃん」
→ 保守層：「そもそもコピペに興味ない」
→ 結果：ポジション設計として中途半端
```

### 解決策
- 「人間設計 vs AI生成」の対立軸ではなく **「AI活用のための人間設計ライブラリ」** として再定義する
  - 例：「AIへの指示テンプレとして使える」「AIと組み合わせると最速」
- 「人間が設計した」は **品質の根拠** として使い、それ自体を主訴求にしない
- AI文脈で語るなら「AIの出力精度を上げるための素材」という切り口が今後の主流になる

## 否定6：ペルソナ分析の前提そのものが危うい

### なぜ失敗するか

元データ（リベシティつぶやき）には以下の構造的な偏りがある：

1. **発信バイアス** — 「つぶやく人」はリベシティ内の少数派。実際の大多数（ROM専）の声は含まれていない
2. **ポジティブバイアス** — 「うまくいった・慣れてきた」系の報告が多い。挫折・離脱した人は投稿しない
3. **期間バイアス** — 直近3ヶ月は特定のAIブームの影響を受けている可能性があり、通常期とは異なるかもしれない
4. **検索バイアス** — 「swell」で検索しているため、SWELLに肯定的な投稿しか拾えていない。批判・不満・諦めた声がない

### 解決策
- ペルソナ分析は **「仮説」として扱い、施策のたびに検証する**
- リベシティ内で直接ヒアリング（スキル依頼・コメント）してリアルな声を集める
- 実際に発信してみて、反応率・コメント内容から仮説を修正する **ABテスト発想** を持つ
- 「刺さると思った角度で反応がなかった」を失敗ではなく **データとして蓄積する**

---

## まとめ：全否定した上で残る「本当に使える訴求」

すべて否定した後に残るのは以下の2点のみ。

### 残る訴求① 「体験談ベースの等身大発信」
- 分析・Tips・宣伝より先に **「自分がSWELL-DPを作った背景・失敗・学び」** を共有する
- 「このツールを作った人」の存在が見えてはじめて、ツール自体が信頼される

### 残る訴求② 「AI×SWELLの実践者向け・具体的なユースケース」
- 「AIでSWELL実装をしている人向け」に絞り、**「このコードをAIに渡したらこうなった」** という具体シーンを見せる
- 広く届けようとするほど薄くなる。最初は **最もニーズが明確な1タイプだけ** に刺さる発信をする

---

> **結論：ペルソナ分析の「刺さる」は仮説に過ぎない。**
> 実際に発信して、反応がなかった理由を分析することが唯一の正解への道。
> この資料は「失敗する前に読む」ではなく「失敗した後に照合する」ための資料でもある。
